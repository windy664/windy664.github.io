我给 Minecraft 服务器写的 AI Agent 跑在 Velocity 代理端，要回答「这台子服有什么命令能用」。第一版设计看起来很合理，实机一跑，它对着子服的 464 条命令，**连猜了九次关键词，撞满迭代上限，最后什么也没答出来**。这篇讲我怎么把它从「每次现查」改成「预同步本地查」，以及背后那条我觉得做 Agent 工具时最该记住的原则。

<!-- more -->

## 第一版：让 Agent 实时去子服「现查」

需求是：中心 Agent 要知道某台子服真实装了哪些命令（不同子服插件不同）。最准的数据源不是文档，是服务器自己——Bukkit 的 `CommandMap` 里就存着当前实际注册的每条命令。

于是我做了个工具 `list_commands_on_server`：Agent 调它 → 中心经总线把请求发到子服 → 子服现扫 `CommandMap` → 把结果传回来。听起来挺直接。

## 实机：它开始猜

日志（我后来才补上的工具结果日志，下面会说为什么这很重要）长这样：

```
Tool call: list_commands_on_server {"server":"earth","query":"cmi"}
Tool call: list_commands_on_server {"server":"earth","query":"spawn warp fly speed"}
Tool call: list_commands_on_server {"server":"earth","query":"effect enchant summon"}
Tool call: list_commands_on_server {"server":"earth","query":"give tp gamemode"}
Tool call: list_commands_on_server {"server":"earth","query":"money"}
...（共 9 次）
已达到最大迭代次数（10），任务未完成
```

问题暴露得很彻底：

1. **每一次猜词，都是一整轮「中心 → 总线 → 子服 → 现扫 → 回」的跨进程往返。** 一次往返只够查一个关键词。
2. 用户问的是「有什么**好玩的**指令」——这是个语义问题。命令名里没有「好玩」二字，关键词匹配不到，于是 Agent 只能不停换词试。
3. 464 条命令是个大数据集，Agent 没有全局视野，只能盲猜。九次往返之后，迭代上限到了，任务失败。

更要命的是，我一开始**连工具返回了什么都没打日志**——只记了「调用了哪个工具」。所以第一次排查时，我根本看不见 Agent 在猜什么、拿到了什么，只看到它反复调同一个工具然后超时。补上「工具入参 + 返回」的日志后，上面那串才显出来。**可观测性不是锦上添花，是排查的前提。**

## 病根：per-query live 模型本身就是错的

我盯着日志想了很久，结论是：让 Agent 在一个大数据集上「每次提问现查、且查询要跨进程往返」，是个结构性错误。

- 跨进程往返有延迟，一次只够查一点点；
- Agent（一个 LLM）面对模糊/语义查询，本能就是反复试不同关键词；
- 二者一叠加，就是 N 次往返 × 延迟 = 组合爆炸，外加把迭代预算烧光。

真正该做的，是把数据**预先同步到中心、在本地查**——Agent 一次本地检索就拿到全局，零往返。

## 第二版：子服建目录 → 推回中心 → 本地检索

新架构是这样：

```
子服启动 / 插件变更
  └─ 扫 CommandMap 建「能力目录」(命令+别名+描述+来源)
       └─ 经总线主动推回中心
            └─ 中心 CapabilityRegistry 收齐各子服目录
                 └─ Agent 用 search_capabilities 查【中心本地】索引（零往返）
```

几个关键决定：

- **推送时机是「变更驱动」**：子服把目录建好后算一个内容签名，存在本地文件里；下次启动或插件增删改时比对签名，**变了才推，没变就不推**。稳态下零推送。
- **中心持久化**：收到的目录落盘，重启直接从盘载入，不依赖子服重推。这两点是配套的——正因为中心有「永久记忆」，子服「没变就不推」才安全。
- **渐进披露**：无关键词时返回「按来源插件分组的概览」（`Essentials 45 条、CMI 120 条…`），而不是平铺 464 条；有关键词才展开。绝不把上千条塞进 LLM 上下文——**上下文 token 才是真瓶颈**。

改完之后，同样的问题，日志变成一次 `search_capabilities` 本地命中，秒回。

## 我学到的

> 一个好的 Agent 工具，应该**快、有界、可消化**地返回结果。

具体拆开：

- **能预同步的，别让 Agent 现查。** 数据相对稳定（命令表、文档、目录），就提前同步建索引，Agent 查本地。把「实时性」让位给「可检索性」。
- **跨进程往返要省着用。** 每多一次往返，就给 Agent 多一次「猜歪了再来」的机会。
- **永远先返回摘要，按需下钻。** 大数据集平铺进上下文，既烧 token 又让 Agent 抓不住重点。
- **没有日志就没有真相。** 工具的入参和返回都要能看见，否则你只能盯着「超时」干瞪眼。

最后那条「能力目录预同步」的思路，其实是被用户一句话点醒的：「你不就是在 Bukkit 端建好一份清单、有变更才报给上游、上游自己存一份永久记忆吗？」——是的，就是这样。有时候最好的架构，是别让聪明的模型去干本该提前算好的事。
