定下了 WindyAgent 的技能机制（[服主写、Agent 按名调用](/2026/06/15/skill-system-not-llm-eval/)），接着冒出一个更难的问题：技能到底**存在哪、在哪执行**？

WindyAgent 有两种形态：一种是用一个**中心节点**（跑 Agent 大脑的群组代理）统管底下若干**子服**；一种是单台服务器自己跑 Agent。技能放哪，第一版我图省事全塞进了子服。后来想统一管理，才发现这中间卡着一个绕不开的物理约束。

<!-- more -->

## 那个绕不开的约束

脚本技能是 Groovy，里头调的是子服那台机器上的 **Bukkit / 插件 API**（在线玩家、经济、世界状态……）。而中心节点是个代理层，**它的 JVM 里根本没有这些 API**。

一句话：**脚本技能在中心跑不起来，物理上只能在子服执行。** 你没法把它「挪到中心」去跑。

但反过来，**纯文字（流程）技能**只是给 Agent 大脑的一段指令——而大脑就在中心。它在中心存、在中心用最自然，塞到子服反而别扭。

于是「技能放哪」根本不是二选一，而是**按类型劈开**：

| | 存哪 | 在哪执行 |
|---|---|---|
| 文字技能 | 中心 | 中心（Agent 直接读正文，用现有工具办事） |
| 脚本技能 | 中心（权威源） | 下发到子服执行 |

## 最终形态：中心权威库 + 下发

- **中心持有唯一权威技能库**：所有技能在这里写、在这里管，单一事实来源。
- **文字技能**：在中心存、在中心直接挂成 Agent 的工具，调用即返回正文给大脑。
- **脚本技能**：在中心存源码，子服一连上来就**下发**到它的 `skills/` 目录；真正执行还是子服那套 GroovyShell + 主线程引擎。中心通过一个 `run_skill_on_server` 远程工具去调它。

「统一在中心管」成立了，又没违反「脚本只能在子服跑」——**管理收归中心，执行留在能执行的地方。**

## 一个差点搞错的细节：能力目录里的 server 绑定

中心的 Agent 怎么知道「这个脚本技能要去哪台子服调」？这靠**能力目录**——每台子服把自己有哪些能力上报给中心，中心检索时每条都带着「来自哪台服」，于是 `run_skill_on_server` 能填对目标。

我一度想让**中心**直接发布脚本技能的目录条目。错了：中心不知道该把它绑到哪台服。正确做法是让**子服上报自己实际持有的技能**——下发过去之后，子服把它纳入自己的目录再上报，server 绑定天然就对了。

所以下发链路要闭环：

```
子服上线 → 中心收到它的能力目录(此刻还没那个技能)
        → 触发下发：把命中它的脚本技能 push 过去(写文件)
        → 子服 reload + 重建目录上报
        → 中心收到新目录，这下 run_skill_on_server 找得到了
```

我特意在子服侧加了个回调：收到下发后**主动重建一次能力目录**，让新技能尽快能被搜到，不用干等下一次插件事件。

## 默认技能「谁来释放」也得分清

还有个小设计：全新部署时，技能库该自带几个默认技能（不然 Agent 一上来一身能力都没有）。「谁释放」要按角色分：

- 中心 / 单机：自己就是技能库的归属，**首启从内置资源释放默认技能**。
- 子服：**不释放**——它的技能是中心下发来的，自己别瞎生成，免得和中心权威库打架、出现谁覆盖谁的歧义。

## 收获

这次最大的体会：**架构不是你想怎么优雅就怎么优雅，得先认物理约束。** 「技能全放中心统一管」听着干净，但「Bukkit API 不在中心」这个事实，直接决定了脚本必须留在子服。认了它，剩下的「中心权威 + 下发」反而水到渠成。

做 Agent 这类「大脑在一处、手脚在多处」的系统时，很多纠结本质都是同一个：**先想清楚「这件事物理上只能在哪做」，再谈怎么调度。** 把 Agent 的「决策」和「执行」分层，决策集中、执行就近——大概是这类分布式 Agent 绕不开的骨架。
