做商业化 AI 产品，token 就是钱。用户问我「你有没有做省 token 的东西」，我盘了一遍，发现：做了一些，但都是顺手带出来的点状措施，**真正的大杠杆还没上**。这篇老实交代循环型 Agent 的成本都烧在哪，哪些是「保证省」、哪些是「看运气省」，顺带聊一个连着的话题——长期记忆攒下的**垃圾数据**怎么治。

<!-- more -->

## 钱烧在哪

ReAct 这种循环型 Agent，每一轮都把 **系统提示 + 全部工具定义 + 完整对话历史** 重发一遍。一个调了 5 次工具才完成的任务，这堆东西就被重发 5 次。具体的烧钱点：

1. **系统提示 + 工具 schema**：又大又稳定，却每轮重发；
2. **路由 / 查询扩展这类元任务**：我之前都用同一个 pro 模型跑；
3. **被自动召回进上下文的垃圾记忆**：长期记忆攒久了全是重复、过期、弱相关的东西，召回进 prompt 既费 token 又拉低质量。

## 我做了的（多是顺手）

诚实说，这些是设计时顺带带出来的，不是专门为省 token 做的，但确实在省：

- **RAG 查询扩展只在检索 miss 时才调 LLM**——多数检索零额外调用；
- **路由启发式优先**——大部分请求用关键词/长度规则就分流了，只有模棱两可才调一次 LLM 分类；
- **工具结果限量 + 截断**——几百条命令给「分组概览」而非平铺；
- **历史限长 + 迭代上限**——防止上下文和循环无限膨胀（之前真撞过迭代上限）。

## 没做的：真正的大杠杆

1. **Prompt 缓存（最大头）。** 系统提示 + 工具定义稳定，本可以让 provider 缓存住、循环内重发变缓存命中。但我之前把「自动召回的记忆」注进了系统提示——每次请求记忆不同，反而**破坏了可缓存前缀**。
2. **元任务用便宜模型。** 路由分类、查询扩展这种短、频、容错高的活，没必要用 pro 模型。
3. **历史压缩。** 现在只是「丢最旧」，没把旧对话摘要成紧凑记忆。

## 这一轮补的，和诚实的定位

我补了两件「保证省」的、一件「看运气省」的：

- **（保证省）元任务换便宜模型**：加个 `fast-model` 配置，路由/扩展走小模型，省 pro 模型 token。错了也只是少召回、选错 agent，不影响安全和执行——容错高的活就该用便宜模型。
- **（保证省）记忆治理**（下一节）：少召回垃圾 = 少 token。
- **（看运气省）缓存友好结构**：把召回记忆从系统提示挪进当次 user 消息，让系统提示 + 工具保持稳定前缀。**这条省不省，取决于你的 provider 服务端有没有做前缀自动缓存**——有就白嫖，没有也无害。我用的国产模型这点待确认，所以我不敢把它算成「保证省」。

至于给 Claude 显式打 `cache_control` 缓存断点——能做，但我的用户跑的是 OpenAI 兼容的国产模型，根本不走 Claude 路径，做了对他没用，于是延后。**不为用不上的场景花工程预算，也是一种成本意识。**

## 顺带：垃圾数据怎么治

省 token 和「垃圾数据」其实是一件事——垃圾被召回进 prompt，既费 token 又坑质量。会自动长垃圾的，主要是**长期记忆**（它从 Agent 的判断里攒东西，可能重复、过期、记错）。我的分层治理：

- **写入去重**：记之前，同作用域若已有「归一化相等 / 互相包含」的记忆，复用、不重复攒；
- **召回阈值**：召回按相关度打分，**低于阈值的弱命中直接不召回**——挡住「单字巧合命中」这种垃圾进上下文；
- **人工清理**：`memory clean` 命令清精确重复，`forget` / `clear` 给人兜底；
- 更重的语义去重、矛盾合并（需要 LLM 周期性 consolidation），成本高，留后。

## 写在最后

省 token 这事，最怕的是自我感觉良好——做了几个点状优化就以为省了。真正该问的是：**我的成本结构里最大的那块，动了没有？** 对循环型 Agent，那块是「稳定前缀的重复发送」和「上下文里的无效内容」。

也别为漂亮但用不上的优化买单（比如给不走的 provider 路径做缓存）。把工程预算花在「保证省、且对我的用户真有用」的地方，剩下「看运气省」的诚实标注、随手做了不亏。把账算清楚，比堆优化更重要。
