WindyAgent 的中心（Velocity）要知道「现在有哪些子服可以指挥」。我一开始随手就用了能力注册表的 `servers()`——结果点一个早就关掉的子服，命令发过去**白等 5 秒超时**。这个 bug 逼我把两个一直被我混为一谈的概念拆开：**「它能做什么」和「它此刻在不在」，根本不是一回事。**

<!-- more -->

## 起因：一个「为了方便」的持久化

子服启动时会把自己的能力目录（几百条命令）推给中心，中心存进一个 `CapabilityRegistry`，Agent 靠它本地检索「哪个服能跑什么命令」，零总线往返。

为了「重启中心不用等子服重推」，我给注册表加了**持久化**——目录落盘，中心重启时 `load()` 直接读回。这个优化本身没错。

错在我后面图省事，把「当前有哪些子服」也用了同一个 `registry.servers()`：

```kotlin
connectedServers = { registry.servers() }   // ← 看着对，其实是坑
```

`registry.servers()` 返回的是**「曾经见过的所有子服」**——因为它从盘上读，包含早就离线的。WebUI 的子服选择器列出来一堆，里面混着关掉的；Agent 派命令时也照着这个列表发，发给一个离线的服，只能干等到超时。

## 想清楚：我缓存的到底是什么

复盘的时候我才意识到，我把两种性质完全不同的信息塞进了一个容器：

| 信息 | 性质 | 能不能缓存/持久化 |
|---|---|---|
| 「子服 earth 有哪些命令」 | **能力**，运行期基本不变 | 能，而且应该缓存（重启免重推） |
| 「子服 earth 此刻连着吗」 | **在线状态**，每秒都可能变 | **绝对不能缓存** |

`registry` 缓存第一种是对的。但「在不在」是**实时状态**，它天然就该问「此刻真的握着连接的那一方」，而不是问一个会落盘的注册表。**持久化的便利，悄悄污染了「实时性」的语义。**

## 修法：让总线给出「权威在线集」

谁最知道「谁此刻连着」？是总线本身——我那个自建的 Socket 中枢，手里攥着每条活动连接，子服 `REGISTER` 时加入、断开时移除。这才是在线状态的**唯一可信来源**。

于是我给总线抽象加了一个方法：

```kotlin
/** 当前真实在线的子服名集；null = 该传输报不了，调用方自行退回。 */
fun onlineServers(): Set<String>? = null
```

- Socket 中枢：返回当前活动连接集（权威、实时）。
- 报不了的传输（比如没有心跳的 Redis 模式）：返回 `null`。

中心侧的取值就变成：

```kotlin
val online = { bus.onlineServers() ?: registry.servers() }
```

能判定就用实时集；判定不了（`null`）才退回注册表那份「曾见过」——这样在能做心跳的传输上立刻准确，在做不到的传输上也不退化。

关键是边界划清了：**能力检索仍然走注册表**（「曾见过 earth 有什么命令」，离线也无所谓，反正只是查它会什么）；**「在不在 / 能不能派」改用实时集**。两个概念，两个数据源，各管各的。

## 更普遍的一条教训

这个 bug 小，但它戳中的东西不小：**缓存「它是什么」通常没问题，缓存「它现在怎么样」几乎总是错的。**

身份、能力、配置——这些相对静态的，缓存/持久化是优化。在线、健康、负载、连接——这些每刻都在变的，一旦缓存，你就是在拿「过去某一刻的快照」冒充「现在」。`servers()` 这个名字本身也有锅，它听起来像「现在的服务器」，其实是「历史上见过的服务器」。

现在我加任何一个「列出 / 判断当前状态」的接口前，都先问自己一句：**这个数我是从「此刻真握着它的那一方」拿的，还是从某个图方便的缓存里拿的？** 一字之差，就是「准」和「假在线」的区别。
