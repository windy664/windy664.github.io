MCP（Model Context Protocol）火了之后，几乎每隔一段时间就有人问我同一个问题：「你那个 Minecraft 运维 Agent，中心大脑要控制一堆子服，为什么不把**每个子服都做成一个 MCP Server**？中心当 MCP Client 一连，工具自动发现、协议还标准，多干净？」

这个问题问得非常好——好到我第一反应也是「诶，好像有道理」。但当我把它放进我们真实的部署拓扑里走一遍，结论反过来了：**在我们这个场景，把子服做成 MCP Server 是逆流而上**。它会把一个「子服零入站」的架构，拆成「N 个入站攻击面 + N 份重复的安全逻辑」，换来的标准化收益我们在内部又用不上。

这篇文章我想把这个决策**完整地**推一遍——不是「MCP 不好」，MCP 很好，我们自己就是 MCP 的消费方；而是想说清楚一件更普适的事：**协议标准化是有位置的，它属于边界，不属于内部总线**。顺带把「那到底该在哪儿用 MCP」也给出一个我认为正确的答案。

## 先把话说清楚：MCP 是什么，我们的架构又是什么

聊「该不该用」之前，得先让两边站在同一个坐标系里。

### MCP 的心智模型

MCP 本质是一个 **Client ↔ Server 的 JSON-RPC 协议**，让「一个带 LLM 的宿主」去发现并调用「一个工具提供方」暴露的能力。它规定了几件事：

- `tools/list`：Server 告诉 Client「我有哪些工具、每个工具的 JSON Schema 长啥样」；
- `tools/call`：Client 说「调用工具 X，参数是这些」，Server 执行并回结果；
- 还有 `resources`、`prompts` 等，但对我们这个讨论，核心就是上面这对「列能力 + 调能力」。

关键在**传输层**。MCP 的官方传输只有两类：

1. **stdio**：Client 把 Server 当**本地子进程**拉起来，用标准输入输出通信。适合「本地工具」，比如你在 Claude Desktop 里挂一个读本地文件的 server。
2. **HTTP + SSE（streamable HTTP）**：Server 是一个**网络服务**，监听端口，Client 通过 HTTP 连过去。适合「远程/常驻服务」。

记住这两个选项，等下是全篇的胜负手。

### 我们现在的架构：中心大脑 + 总线 + 能力提供方

我们的运维 Agent 是这么分布的：

```
                    ┌─────────────────────────────┐
                    │      中心「大脑」(Velocity)   │
                    │   LLM + 工具 + 审批/审计门控   │
                    └──────────────┬──────────────┘
                                   │  MessageBus
                    ┌──────────────┼──────────────┐
                    │              │              │
              ┌─────▼────┐   ┌─────▼────┐   ┌─────▼────┐
              │  子服 A   │   │  子服 B   │   │  子服 C   │
              │ Capability│   │ Capability│   │ Capability│
              │  Handler  │   │  Handler  │   │  Handler  │
              └──────────┘   └──────────┘   └──────────┘
```

中心大脑（跑在 Velocity 代理上）持有 LLM 和全部工具。要在某个子服上干活时，它经一条 **MessageBus** 把动作下发到那个子服，子服里的 `BukkitCapabilityHandler` 接住、在本机执行、把结果回传。

这条总线的传输是**可插拔**的，目前有三种实现：

- `inprocess`：同 JVM 直调（单机嵌入式）；
- `socket`：子服连到中心开的 socket；
- `redis`：所有节点通过 Redis 的发布订阅通信。

中心侧每个「远端能力」都被包装成一个普通的 `AgentTool`，比如 `run_command_on_server`：

```kotlin
class RemoteCommandTool(
    private val bus: MessageBus,
    private val timeoutMs: Long,
    private val guard: CommandGuard,
    private val audit: AuditLog,
    private val pending: PendingApprovals
) : AgentTool {
    override fun execute(toolCallId: String, inputJson: String): ToolResult {
        // …先过安全护栏（高危 + 不可信 → 拒；高危 + 可信 → 入审批闸）…
        val reply = bus.dispatch(server, "run_command", argsJson, timeoutMs).get(...)
        return ToolResult.success(toolCallId, reply.content)
    }
}
```

子服侧的 `BukkitCapabilityHandler` 就是一个大 `when`，把动作映射成本机操作：

```kotlin
private fun dispatch(req: ToolRequest): ToolReply = when (req.action) {
    "run_command" -> ToolReply(req.requestId, ok, actions.executeCommand(cmd))
    "fs_write"    -> /* 写文件 + git 提交 */
    "health_query"-> ToolReply(req.requestId, true, healthSnapshot())
    // …
}
```

**先记住一个观察**：这套东西的形状，跟 MCP 高度相似——`BukkitCapabilityHandler` 就是 `tools/call`，另有一个 `CapabilitySync` 把每个子服的能力目录推回中心，那就是 `tools/list`。我们其实**已经有一套 MCP 形状的协议**了，只是没叫它 MCP。这一点等下很重要。

## 提议长什么样：子服 = MCP Server

把提议具体化。如果每个子服变成一个 MCP Server，图会变成这样：

```
        ┌───────────────────────────┐
        │     中心大脑 (MCP Client)   │
        └───┬─────────┬─────────┬────┘
            │ HTTP/SSE│         │
        ┌───▼───┐ ┌───▼───┐ ┌───▼───┐
        │子服A  │ │子服B  │ │子服C  │
        │MCP Srv│ │MCP Srv│ │MCP Srv│
        │:8931  │ │:8931  │ │:8931  │
        └───────┘ └───────┘ └───────┘
```

每个子服的插件里内嵌一个 HTTP+SSE 的 MCP Server，监听一个端口；中心大脑作为 MCP Client，分别连上去，`tools/list` 拿工具、`tools/call` 干活。

听起来确实标准、解耦。但魔鬼在**那几根箭头的方向**。

## 第一刀：拓扑。子服凭什么开一个入站端口？

这是压垮提议的**第一块、也是最重的一块石头**。

MCP 的远程传输是 HTTP+SSE，意味着**子服必须监听一个端口、等中心连进来**。也就是说，数据流是「中心 → 子服」的**入站（inbound）**连接。

再看子服在现实里的位置：**它挂在 Velocity/BungeeCord 代理后面，通常在 NAT 或防火墙内网里**。很多人的部署是「一台机器跑代理 + 好几个子服进程」，或者「子服在容器/内网，只有代理有公网」。这种拓扑下，「让中心连进每一个子服」意味着：

- 每个子服要**对外开一个端口**；
- 要给每个端口配**鉴权**（否则谁都能连进你的 MC 服执行命令）；
- 跨机器时要处理 **NAT 穿透 / 端口映射 / 防火墙白名单**；
- N 个子服 = **N 个入站端点** = N 个要维护、要防守的洞。

而我们的总线是反过来的——**监听方不是子服**。这里得先说句实在话：**任何联网传输都得有人开端口，「零端口」不存在**；连 Redis 自己都得监听 6379。所以真正该问的不是「开不开口」，而是三件事——**开几个、开在哪台机、谁来守**。看 `redis` 传输：所有节点只需要「出站」连到 Redis，走发布订阅互相收发；**子服自己一个入站口都不开**，一个在 NAT 后面、只有出站网络的子服零额外配置就能接入。那唯一的监听口在 **Redis 那台**，不在子服上。

```
MCP-per-子服：中心 ──inbound──▶ 子服:8931   （开 N 个口，还偏偏开在最该藏的子服上）
socket 总线：  子服 ──dial-home──▶ 中枢:port  （开 1 个口，开在本就公开的代理/中枢上）
redis 总线：   中心 ──▶ Redis:6379 ◀── 子服   （开 1 个口，开在 Redis；两端都只出站）
```

这一条，几乎就足够定案了。注意别把它读成「总线零端口、MCP 才开口」——**不是**。准确说是：**总线只开 1 个、开在本来就对外的中枢/broker 上，子服开 0 个；MCP 要开 N 个、还开在最该藏的子服上。** 位置尤其致命：中枢就是玩家在 25565 上连的那台代理，本就该暴露，在它上面多一个带 secret 的内网口是**边际成本**；而子服恰恰是代理架构**存心藏在后面**的机器，玩家从不直连它——往子服上开入站口，捅破的正是这道边界。对一个「AI 能在里面执行任意命令、还能改文件」的系统来说，多开 N 个这样的口是我最不想要的方向。

有人会说：那用 stdio 传输不就没有端口问题了？但 stdio 要求 Client **把 Server 当子进程拉起来**——中心大脑在 Velocity 进程里，它没法把一个跑在**另一台机器**上的 MC 服务器当子进程。stdio 只适合同机本地工具。所以远程子服只能走 HTTP，绕不开入站端口。

### 澄清一个容易读岔的点：「零暴露」是 socket 和 redis 共有的，不是 redis 独占

这里得赶紧补一句，免得被误读成「所以该全走 redis」。前面我举 Redis 发布订阅当「零暴露」的例子，但**「子服零暴露」这个性质，来自「子服主动连出」，而不是「用了 Redis」**。我们的 `socket` 传输同样满足：中枢（`SocketHubBus`）开监听，子服（`SocketClientBus`）**dial-home 主动连进来**、断线自动重连。子服在 socket 模式下**一样一个端口都不开**。

所以**这一刀砍的是 MCP，不是 socket**。socket 和 redis 在「暴露面」这条轴上是**打平**的，都吊打「子服各开一个入站 HTTP 口」的 MCP 方案。

那 socket 和 redis 差在哪？差在**另一条轴——基建成本 vs 规模解耦**，跟安全无关：

- `socket`：**零外部依赖**，中枢开个带 secret 的端口即可，星型拓扑。适合绝大多数「一个代理 + 几个子服，常在一台机/一个内网」的部署。这也是我们的**默认** transport。
- `redis`：要**额外跑一个 Redis broker**，但换来解耦——多机、大量节点、动态增减、中枢自己也在 NAT 后（两边都出站连 broker）、中枢高可用。规模上来了才划算。

而且「全上 redis」在安全上**并不是净赚**：你只是把「中枢监听一个带 secret 的端口」换成「你得自己跑并守好一个 Redis 实例」——而裸奔的 Redis 是出了名的被扫打目标。按本文那条「收敛暴露面/信任点」的原则，小规模下 socket 反而**更收敛**（少一个要防守的常驻服务）。所以我们**两个都留**：`socket` 兜底、`redis` 应对分布式，按拓扑选，而不是「谁更优」。

## 第二刀：信任门控会被迫下沉，然后复制 N 份

第二块石头是安全模型。

我在 [给会动手的 Agent 上护栏](/2026/06/13/guardrails-for-acting-agent/) 里写过，我们所有的危险动作都过一道 `CommandGuard`：高危命令 + 不可信来源直接拒，高危 + 可信来源进 `PendingApprovals` 审批闸，每一步进 `audit.log`。**关键是这道门控住在中心**，子服只负责执行「已经被中心 gate 过的、来自可信总线的」动作。

这套设计的前提是：**总线是一条可信通道**。子服可以信任「总线上来的请求都是中心审过的」，所以它不用自己再判一遍权限。整个系统只有**一处**需要做信任判定。

现在把子服变成 MCP Server 会发生什么？MCP Server 天然是「谁连上来都能调 `tools/call`」的。子服**不能再假设调用方可信**了——任何能连到它端口的 MCP Client 都能让它执行命令。于是安全逻辑被迫**下沉到每一个子服**：

- 每个子服都得自己实现鉴权（谁能连）；
- 每个子服都得自己实现审批闸（这条命令要不要人工确认）；
- 每个子服都得自己记审计；
- 而且这些逻辑**在 N 个子服上重复 N 份**，还要保持一致——改一次策略要同步 N 处。

这是典型的**把「单点信任边界」摊成「N 点自我防御」**。安全工程里这几乎总是错的方向：你想要的是**收敛**攻击面和策略点，不是发散。我之前做 [32 项安全自查](/2026/06/18/agent-security-audit-32-issues/) 的时候最深的体会就是——**能少一个信任边界，就少一整类漏洞**。

总线模型是「一条可信通道 + 中心统一门控」，这正是我要的收敛。MCP 化把它打散了。

## 第三刀：我们已经有 MCP 形状的协议，重写只为标准化

回到前面那个观察：`BukkitCapabilityHandler`（列能力 + 调能力 + 回结果）本质就是 `tools/list` + `tools/call`。我们**已经有一套能跑的、经过实战的私有协议**。

那把它换成 MCP 能换来什么？**标准化**——工具自描述、schema 规范、生态里的 client / inspector 都能用。这在**对外**是真价值。但在我们**内部**，两端都是我们自己写的：中心是我们的，子服也是我们的，中间协议叫什么名字对功能没有区别。为了「标准」这个在内部用不上的收益，去重写整条传输管线，还要在 Bukkit 插件里塞一个 MCP Server 运行时——**这是纯成本，没有对应的内部收益**。

工程上有个朴素判断：**当协议的两端都归你，且你已经有能跑的实现时，换协议的唯一正当理由是「有第三方要接这个边界」**。子服和中心之间没有第三方。所以这里没有换的理由。

（顺带一提，这也呼应我之前写的 [一个扩展点收口所有工具来源](/2026/07/06/tool-contributor-single-extension-point/)：加能力应该是「往贡献者清单加一项」，而不是「引入一套新协议栈」。）

## 第四刀：Bukkit 主线程与生命周期税

第四块石头偏工程细节，但很实在。

Minecraft 的 Bukkit API **绝大多数必须在主线程调用**——我在 [混合端的主线程与调度器](/2026/06/13/hybrid-server-mainthread-and-scheduler/) 里详细写过这套约束。我们现在的 `BukkitActions.onMainGuarded` 已经把「总线来的异步请求」安全地跳回主线程执行，还附带一道超时看门狗。

如果在插件里内嵌一个 HTTP+SSE 的 MCP Server，你并没有省掉这套复杂度——你**还是**得：

- 自己管 HTTP server 的线程池；
- **还是**得把每个 `tools/call` 跳回主线程执行；
- 额外再背上 HTTP server 的**生命周期**（绑定插件的 enable/disable、优雅关闭）、**端口冲突**、**TLS 证书**、SSE 长连接的重连……

也就是说，MCP 化没有消除任何既有复杂度，只是**在上面又叠了一层 HTTP 服务的运维税**。而这层税，我们用总线是不交的：子服连出去就完了，没有 server 要托管。

## 我得替对方说话：MCP 什么时候确实赢？

到这儿你可能觉得我在单方面锤 MCP。不。我特意要把对面的理由摆正——不然这不叫辩论，叫立靶子。

MCP 在这些场景是**明确更优**的：

1. **工具要给第三方 / 外部宿主用。** 如果你希望 Claude Desktop、Cursor、或者别人写的 Agent 都能驱动你的能力，MCP 是通用语言，私有协议做不到。
2. **接入外部现成工具。** 反过来，社区已经有一堆 MCP Server（文件系统、Git、数据库、浏览器……），你的 Agent 直接消费它们，比自己写集成快得多。
3. **强解耦 / 独立演进。** MCP Server 可以独立部署、独立发版、用任意语言写，跟宿主完全脱钩。
4. **自描述与可发现性。** schema 化的工具定义，配套的 inspector / 调试器，生态成熟。

事实上——**我们自己就是 MCP 的消费方**。我们的 Agent 有 `McpLoader` / `McpToolAdapter`，配置里有一段 `mcp.servers`，就是用来把外部 MCP Server 暴露的工具喂进 Agent 的：

```yaml
# windyagent-config.yml
mcp:
  servers:
    - name: some-external-tool
      url: https://...
      headers: { Authorization: "Bearer ..." }
```

所以我的立场从来不是「MCP 不好」。而是：**MCP 是边界协议，不是内部总线。** 上面四条优势，全都发生在「你的系统」和「别人的系统」之间那道**边界**上。而中心大脑到自家子服，是**内部**，没有边界，自然也用不上边界协议的好处，却要吃它全部的成本。

## 那到底该在哪儿用 MCP？——在「脑」，不是每个「手脚」

真正有意思的是：如果我确实想让外部工具链（Claude Desktop、别的 Agent）来驱动我的服务器运维，**正确的落点不是把每个子服做成 MCP Server，而是把中心大脑做成一个 MCP Server**。

因为中心大脑**本来就经总线聚合了所有子服的能力**。把它——且只有它——暴露成一个 MCP Server：

```
   外部 MCP Client (Claude Desktop / 别的 Agent)
                    │  一个端点、一套鉴权
                    ▼
        ┌───────────────────────────┐
        │   中心大脑 = 单个 MCP Server │
        │   （聚合 + 门控 + 审计）      │
        └──────────────┬────────────┘
                       │  总线（子服仍零暴露）
             ┌─────────┼─────────┐
          子服A       子服B      子服C
```

对比一下两种「MCP 化」：

| 维度 | 每个子服 = MCP Server | 中心大脑 = 单个 MCP Server |
|---|---|---|
| 对外端点数 | N 个 | **1 个** |
| 子服是否要暴露 | 要（N 个入站口） | **不要（仍走总线，零暴露）** |
| 鉴权点 | N 处，重复 | **1 处** |
| 安全门控 | 下沉到每个子服，复制 N 份 | **复用中心已有的** |
| 子服要不要改 | 每个都要塞 MCP 运行时 | **一行不用改** |
| 外部能拿到什么 | 单个子服的能力 | **全部子服的聚合能力** |

结论一目了然：**「中心一个 MCP Server 聚合所有子服」在每一列都不输，多数列完胜「每个子服一个 MCP Server」。** 前者是把 MCP 用在真正的边界上，后者是把边界协议错当成了内部总线。

## 那超大集群呢？——把最硬的反方摆上来

到这儿肯定有人不服：「你说的都是小中集群。等我上到 **10+ 子服、跨多台物理机、生存/创造/小游戏/副本各自独立存档、还要 AI 自动运维**，每个子服单独做 MCP 就是对的了吧？」

我不回避，把这套反方**最强的版本**原样摆出来，它通常有五条：

1. **故障隔离**：某个副本服崩了/更新插件，不该打断其他服的 AI；全收拢在中心，子服一异常就阻塞整条工具链。
2. **本地低延迟**：AI 要高频读写当前子服的本地数据（NPC 状态、区块、背包、子服专属模组接口），直连本服比经中心中转快。
3. **权限精细化**：生存服 MCP 只开生存指令、小游戏服只开游戏管理，粒度更细。
4. **水平扩展**：新子服自带 MCP 一启动，Agent 自动发现接入，不用改中心、不用重启。
5. **负载分流**：大量 AI 批量操作（批量建筑、NPC 对话生成）的 CPU 摊到各子服，不压垮代理。

这几条看着很唬人。但把它逐条放进「一个**做得好**的总线」里对照，就会发现**四条是稻草人或总线本就具备，只有一条是真的——而那条的正解也不是 MCP**。

**① 故障隔离——和用不用 MCP 无关，取决于「请求模型」。** 我们的 `bus.dispatch(server, …)` 是**每请求独立 + 带超时**：子服 A 崩了，只让调 A 的那个 future 超时失败，**完全不挡调 B/C**。会「一个子服异常阻塞整条链」的，是那种把所有调用串行焊死的烂实现——那是实现问题，不是总线 vs MCP 的问题。隔离性由「独立请求 + 超时」保证，拓扑换成 MCP 一分不多。

**② 本地低延迟——这条是真的，也是全篇我唯一让步的地方。但答案不是子服 MCP。** 高频、延迟敏感的本地循环（实时 NPC、逐 tick 读区块），中心→总线→子服确实多一跳。可是：

- 对 **LLM 驱动**的操作，推理本身就是几百毫秒起，LAN 上总线一跳是亚毫秒，**中转延迟在噪声里**，谈不上瓶颈；
- 真有高频确定性逻辑，正解是**让 Agent（或一段本地技能脚本）直接跑在那台子服上**——我们本来就有 **embedded/standalone 模式**：Agent 嵌在子服进程里，本地工具直调，**零往返**。这比「子服开个 MCP Server 等远端脑来戳」**还快**，因为后者仍有一次 client↔server 往返、还得维护一条连接。

所以延迟这条恰恰指向**本地化 Agent**，而不是**子服 MCP**。把智能放到数据旁边，比把数据的入口暴露给远端智能，更对。

**③ 权限精细化——不是 MCP 的专利，反而是它的负债。** 「按子服分权限」中心侧一样做得到（中心知道目标是哪个服，套哪套策略）。而一旦子服变成独立可达的 MCP，它**必须自己再实现一套鉴权 + 白名单**——于是同一套安全策略要在 N 个子服上配 N 份、还得保持一致。第二刀说的「门控下沉、复制 N 份」，在这儿原样复现。

**④ 自动发现/水平扩展——总线早就有。** 新子服 dial-home 连上、经能力目录（`CapabilitySync`）把自己的工具清单推回中心，中心当场发现，**不改中心插件、不重启代理**。这不是 MCP 独享的魔法，是任何带能力目录的总线都有的东西。

**⑤ 负载分流——是个稻草人。** 重活压根不在中心。LLM 推理在模型侧；命令**执行本来就在子服**（总线只把几十字节的 JSON 派过去，子服自己跑）。中心只做路由，CPU 近乎零。「批量操作压垮代理」描述的是一个并不存在的瓶颈。

**还有一处反方自己会绊倒的地方**：这派人通常会补一句「MCP 走内网、不对外暴露端口」。可「不对外」不等于「不开口」——架构里脑要连每个子服的 MCP，那每个子服就在内网**监听一个入站口**，还是 N 个要鉴权的监听。更别说这套的前提是「**多物理机**」——一旦子服在另一台机/NAT 后，「内网直连」的假设当场破产，脑要 inbound 连进每一台，正是最疼的地方。而这恰恰是总线（两端都 dial-home / 出站连 broker）绕开的。**规模越大、越多机，子服 MCP 的入站问题越严重，不是越轻。**

所以我的结论在超大集群不但没松动，反而更硬：**规模上去了，正确形状还是「对外一个 MCP 面（脑）+ 内部走总线 + 延迟敏感处让 Agent 本地跑」**。真正随规模增长的，是「把智能下沉到子服本地」的需求——而那是 embedded Agent 干的事，不是「把子服的入口 MCP 化」。**别把「本地要有智能」错听成「本地要开个 server 等人连」。**

## 澄清一个最容易的误读：不做子服 MCP，不等于子服变哑巴

写到这儿，我猜有人已经憋出一个疑问：「你一路说别把子服做成 MCP、说子服是执行器——那我还要不要调子服的各种工具、读它的数据、改它的文件？子服不就废了？」

**恰恰相反，一样都不少，而且我们现在天天在调。** 这里的关键是把两件事分开：

- **子服 = 能力提供方**：只有它能读自己的文件、跑自己的命令、碰自己的插件——它握着**全部真实能力**；
- **大脑 = 决策方**：它决定「做什么」，子服知道「在自己这台机上怎么做」。

「执行器」不是哑管子，是**握着全部能力、听调度去执行的那一方**。而「用不用 MCP」这场争论，争的**只是把「请在 A 上做 X」送过去的那根管子**，跟「能不能在 A 上做 X」半毛钱关系没有。

举个我们真实的调用——「改子服 A 的 config.yml」：

```
大脑(LLM) 决定：改 A 的 config.yml
   └─ 调工具 write_file_on_server(server=A, path=…, content=…)
        └─ 工具内部：bus.dispatch("A", "fs_write", {…})
             └─ 子服A 的 CapabilityHandler 收到 "fs_write"
                  → 真正写文件 + git 提交 → 回结果
   ◀─ 结果回到大脑
```

同一件事若走「子服 MCP」，无非是 `大脑 ──HTTP──▶ A 的 MCP Server: tools/call fs_write`。**同样的活、同样那个文件被改，能力一分不差**——区别只在那根管子：是「A 主动连出到总线」还是「A 开端口等大脑连进来」。

事实上我们早有一整排这样的远端工具：`run_command_on_server` / `read_file_on_server` / `write_file_on_server` / `delete_file_on_server` / `config_history_on_server` / 查在线 / 查 TPS / 调子服技能……**大脑成天经总线调子服的这些工具、拿它的数据、改它的文件**。不做子服 MCP，这些一个都不会失去。

**一句话：「MCP vs 总线」争的是「用哪根管子」，不是「能不能干」。** 我们要的「调工具、取数据、改文件」是**管子里流的内容**；换不换管子，内容照旧。而总线这根管子，恰好让子服零入站、门控收在中心一处——所以我们留它。

## 提炼成一条原则

如果要把这次决策压成一句能带走的话，我会这么说：

> **标准化协议属于系统的边界，不属于系统的内部。在内部，选能让你「收敛暴露面、收敛信任点」的传输；在边界，选能让「别人接得进来」的协议。**

我们的总线（尤其 Redis 发布订阅）之所以对内部更优，就是因为它同时收敛了两样最贵的东西——**暴露面**（子服零入站）和**信任点**（门控只在中心一处）。而 MCP 之所以对边界更优，是因为它让**外人接得进来**。把这两者摆错位置，你会得到一个「对内暴露 N 个口、对外还是各连各的」的四不像。

所以最终答案，回到最初那个问题——「为什么不把子服做成 MCP？」：

- 因为子服在代理/NAT 后面，**不该各自开入站端口**；总线让它们零暴露就能被够到；
- 因为那会把**统一的安全门控打散、复制到每个子服**，发散攻击面；
- 因为我们**内部已经有 MCP 形状的协议**，重写只为一个内部用不上的「标准」；
- 而 MCP 真正的价值在边界——所以正确的做法是**把「脑」暴露成唯一一个 MCP Server**，让它替所有子服对外说话，子服继续躲在总线后面。

下一步我大概会真的去把「中心大脑 = MCP Server」这条实现掉——让外面的 Agent 也能通过一个口、一套鉴权，驱动整片服务器。到时候再写一篇讲怎么落地。而子服，会继续安安静静地待在总线后面，一个端口都不开。
