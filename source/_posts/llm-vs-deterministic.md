「我们是个 AI 项目」——这句话最大的陷阱，是让你下意识把什么活都塞给大模型。做 WindyAgent 这阵子，我反复在「这步该不该调 LLM」上栽跟头，慢慢摸出一条分界线。

<!-- more -->

## 反例一：让 LLM 逐个估值，14 分钟还算不准

给上千物品估值时，我一度想让 LLM 逐个报价。真跑起来：mimo 给 80 个物品估价，**一批要 170 秒**，5 批接近 14 分钟。而且——

让 LLM 算「A = 4×B + C，B = 4×D + E……」这种十层深的合成树乘法累加，它会算错、会前后不一致。**这是结构化图计算，不是语言任务。** 这种活就该交给确定性代码：我那套配方传播跑完 649 个物品是**秒级**的，而且每次跑结果一模一样、可复现、可解释。

> 一个反直觉的点：分批大小（80）根本不影响总耗时。LLM 估值的耗时≈物品数×单价（mimo ~2 秒/个），跟你切成几批无关。想快，只能**减少要它估的数量**，不是调批大小。

## 反例二：确定性命令会偷偷烧 token 吗？

有人问我：你那些「打字即执行」的确定性命令，背后是不是还偷偷调了 LLM？

不会。处理逻辑是这样的：

```kotlin
val meta = router.dispatch(input, ...)   // 纯代码，看首词匹不匹配命令
if (meta != null) { 回复; return }        // 命中 → 执行返回，根本走不到下面
val resp = agent.run(context)             // ← 唯一调 LLM 的地方
```

输入先过 `dispatch`，首词命中命令（`build`/`status`/`clear`…）就直接执行返回了，**压根进不到 LLM**。确定性命令是**零 token** 那条路。只有「不是命令」的自然语言，才落到 `agent.run` 去花 token。

所以「确定性命令」和「省 token」不矛盾，恰恰相反——它就是为了把高频、规则明确的操作**绕开 LLM**。

## 那 LLM 到底擅长什么

把它干不好的都划走之后，LLM 真正值钱的地方就清楚了：

- **自然语言意图**：「广播一下今晚八点活动」——把人话翻译成工具调用。
- **世界知识**：「下界之星很稀有、基础精华很廉价」——这种判断没法从配方里算，但模型训练时见过。
- **分类**：把一个物品归进「普通/稀有/史诗」哪一档。

第三点是关键的折中。给源头物估值时，我最后没让 LLM 报精确数字，而是让它**只做归档**（它擅长分类），档位对应的金币值写在配置里，精确价靠确定性传播去派生。这样 LLM 调用从「几百个物品」降到「几十个档位判断」，又快又一致。**让 LLM 出标签，让代码出数值。**

## 「只问不做」：弱模型的通病，靠提示词掰，靠命令兜底

还有个坑：你给 LLM 一句祈使「把 Steve 踢了」，弱模型（尤其便宜的）经常**反问「你确定要踢吗」或只口头答应不动手**。我靠系统提示词把它掰过来——明确写「运营指令直接执行，缺参数才追问，别反问要不要做」。

但这是**治标**。治本是：真正高频、要 100% 可靠的操作，做成确定性命令兜底，不赌模型心情。提示词能让它「大概率动手」，命令能让它「一定动手」——两者价位不同，按场景选。

## 一张我自己用的判断表

| 这步任务的特征 | 交给谁 |
|---|---|
| 可枚举 / 可计算 / 要复现 / 高频 | **确定性代码**（命令、传播、规则） |
| 开放语言 / 要世界知识 / 模糊意图 | **LLM** |
| 介于之间（打标、归档、抽取） | **LLM 出标签 + 代码出结果** |

## 写在最后

别因为项目挂着「AI」两个字，就把锤子当成唯一工具，见什么都想敲。LLM 是工具箱里的一把，不是箱子本身。

我现在搭一个功能的默认顺序是：**先问能不能确定性地做掉**——能枚举、能算、能定规则的，全部确定化（顺便零 token、秒回、可复现）；**只在真正需要「理解」和「判断」的接缝处，才请模型出场**。这样的架构最省心也最省钱，而且——把能确定的都确定掉之后，你反而更看得清：LLM 真正不可替代的价值，到底在哪几个点上。
