给 Agent 加 RAG 的过程里，我撞上两个被「RAG 热」糊住的认知误区。一个是「做 RAG 必须有 embedding 模型」，另一个更隐蔽：「上了 RAG，Agent 就靠谱了」。这两个都不对，而且踩坑的代价不一样——前者浪费力气，后者会出事。

<!-- more -->

## 误区一：RAG 必须有 embedding 模型

我的目标用户用的是小米 mimo（OpenAI 兼容）。我兴冲冲要给知识库上向量检索，结果一查 mimo 的模型列表：`mimo-v2.5-pro`、`-asr`、`-tts`……**全是对话/语音/合成模型，根本没有 embedding 模型**。

那一刻我差点以为 RAG 这条路断了。但冷静下来想，RAG 的定义是「**检索 → 把检索结果塞进 prompt → 生成**」，「检索」有好几种实现，向量只是其中一种（稠密检索）：

| 检索方式 | 要 embedding 吗 | 说明 |
|---|---|---|
| 稀疏检索（关键词 / BM25） | 否 | 命令、FAQ 这种关键词强的领域，往往不输甚至更好 |
| LLM 驱动检索 | 否 | 用你**已有的**对话模型做查询扩展/翻译 |
| 稠密 / 向量检索 | 是 | 只有这条需要 embedding |

所以「**严格意义的 RAG，不依赖 embedding**」。向量真正的优势在大段非结构化文本、模糊语义；对一个有界领域（一个服务器的命令、商品、规则），稀疏检索 + 同义词扩展就能拿到大部分价值。

我最后的方案是「**稀疏优先 + 命中不足才用 LLM 扩展查询**」：

1. 先用原查询做关键词检索（中文按二元组切词）；
2. 命中够了就直接返回——**零额外 LLM 调用**；
3. 命中不足，才调一次对话模型把「会员有什么好处」扩成 `会员 vip 权益 福利 member` 这类检索词，再查一遍。

这就是业界叫 「BM25 + LLM 增强检索」的那套，纯靠对话模型补语义，不要任何向量模型。而且因为只在 miss 时才调 LLM，多数查询零额外开销——对一个定价敏感的产品，这点很重要。

中英语义鸿沟（中文「传送」要命中英文 `/tp /home /warp`）我还加了一张结构化的同义词表兜底。Minecraft 命令是有界领域，几十条对照就覆盖了绝大多数，确定、可审计、零成本。

> 结论：**想做 RAG，不必须有 embedding。** 向量是「语义检索」的一种实现，不是 RAG 的同义词。等你真要在大段非结构化文本上做模糊语义召回，再上向量也不迟。

## 误区二：RAG 让 Agent「靠谱」了

这个更要命。我做完 RAG，正美着，用户问了一句：「你不用 RAG 的 agent 以后有没有漏洞风险？」

我才意识到自己把两件事混了。RAG 解决的是「**答得准**」——给回答提供依据（grounding），减少 Agent 凭空编造。但我的 Agent 是会**动手**的：它能在服务器上执行命令、踢人。对一个会执行操作的 Agent 来说：

- **RAG 挡不住「正确检索出来、但很危险」的操作。** 哪怕命令是从真实命令表里查到的，模型理解偏了、或被诱导了，照样能把 `stop`、`op 某人`、`kill @a` 跑出去。
- **RAG 挡不住提示注入。** Agent 会读玩家聊天并据此行动——一个恶意玩家精心构造一句话，就可能诱导它去跑管理员命令。RAG 对这个毫无办法。

一句话点破：

> **RAG 是质量控制，不是安全控制。** 它让 Agent 答得准，但「会动手的 Agent 乱动手 / 被诱导动手」是另一码事。就算你把向量 RAG 上满，这俩洞照样在。

真正该补的，是一层独立的安全护栏：命令分级、来源分权、高危人工审批、操作审计、提示注入加固。这部分我单开了一篇写。

## 写在最后

这两个误区有个共同点：都是被「RAG」这个热词的光环带偏的。RAG 既没有大到「非向量不可」，也没有大到「有了它就万事大吉」。它就是检索增强生成——一个让模型答得更有依据的工程手段，仅此而已。

把它放回它该在的位置，你会发现：没有向量你也能做检索，有了检索你也还得自己管安全。热词不替你思考，工程才替你兜底。
