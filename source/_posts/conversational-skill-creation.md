做 Agent 技能系统时，我遇到一个根本矛盾：**技能本质是代码（Groovy 脚本），但用户是服务器管理员——他们不会写代码。**

传统的解法是「可视化编辑器」「模板库」「拖拽式编排」。但这些方案都有同一个问题：**用户还是要理解「参数」「条件」「循环」这些编程概念。** 对一个只想「每天晚上 8 点给在线玩家发钻石」的服主来说，这些概念是噪音。

<!-- more -->

## 核心洞察：用户描述意图，AI 生成代码

换一个思路：**不要让用户学编程，让 AI 替他编。**

```
服主: "帮我做个技能，每天晚上给在线玩家发5个钻石和100金币"
AI:   [内部生成 SKILL.md + Groovy 脚本]
      → 编译检查 ✅
      → 模拟执行 ✅（会给 2 个模拟玩家各发 5 钻石 + 100 金币）
      → "确认保存吗？"
服主: "确认"
AI:   "已创建技能 daily_gift，说「发每日礼包」即可触发。"
```

用户全程只说中文。SKILL.md 的 frontmatter、Groovy 脚本、参数声明——全是 AI 生成的。

## 技术实现：三个工具 + 一条提示

实现出奇地简单，只需要三个 Agent 工具：

| 工具 | 作用 |
|------|------|
| `create_skill` | 接收 AI 生成的 SKILL.md + Groovy 脚本，落盘到 skills/ 目录 |
| `validate_skill` | 编译检查 + dry-run 模拟执行 |
| `read_skill` | 读取现有技能内容（更新前先读） |

然后在系统提示里写一段指引：

```
当用户想给 Agent 加新能力时，你直接用 create_skill 工具创建：
① 根据用户意图生成完整的 SKILL.md
② 如果需要脚本，同时生成 Groovy 脚本
③ 调用 create_skill 落盘
不要教用户写 SKILL.md 或 Groovy 语法——你替他写，他只需要说人话。
```

**就这么简单。** 没有可视化编辑器，没有模板库，没有拖拽编排。LLM 本身就是最好的「自然语言到代码」编译器。

## 关键细节：验证链

AI 生成的代码不能直接跑。我们加了一条验证链：

1. **编译检查**（`GroovyClassLoader.parseClass`）：只检查语法，不执行任何代码，100% 安全。
2. **Dry-run 模拟执行**：用 mock 对象替代真实 Bukkit API，记录「会做什么」但不真做。
3. **用户确认**：把 dry-run 结果展示给用户，只有确认后才保存。

Dry-run 的 mock 对象设计是关键。`server.getPlayerExact("Steve")` 返回一个 mock Player，`player.inventory.addItem()` 记录「会给物品」而不是真给。用户看到的是：

```
✅ Dry-run 通过
模拟操作：
  1. 🔍 查找玩家 TestPlayer1 → 返回模拟在线玩家
  2. 🎒 给 TestPlayer1 添加 5x diamond
  3. 🎒 给 TestPlayer1 添加 100x gold ingot
  4. 📢 全服广播 "晚安礼包已发放！"
```

## 已有工具优先原则

系统提示里还有一条重要约束：**尽量用已有工具，少写 Groovy 脚本。**

比如「执行一个命令」不需要写 Groovy，直接用 `run_command_on_server` 工具就行。只有现有工具做不到的事（比如调 Vault 经济 API、操作玩家背包）才需要写脚本。

这让大部分技能都是「纯工作流」——一系列 `tool` 步骤的组合，零代码，零风险。

## 对话式修改

改技能也是对话式的：

```
服主: "把金币改成200"
AI:   → read_skill("daily_gift")  读取现有定义
      → 修改 coins 参数默认值
      → create_skill(overwrite=true)  更新
AI:   "已更新，现在发 5 钻石 + 200 金币。"
```

AI 理解「把金币改成200」的语义，定位到 SKILL.md 里的 `coins` 参数，修改后重新保存。用户不需要知道参数在哪里、格式是什么。

## 反思

这个方案的美在于它的**减法**：

- 没有新增任何 UI 组件
- 没有新增任何 DSL 或配置格式
- 没有新增任何编译器或解释器

只是把 LLM 的「理解意图 → 生成代码」能力，接入了现有的技能系统。三个工具 + 一段提示 = 一个完整的「自然语言编程」体验。

**做 AI 产品最反直觉的一点是：最好的功能往往不是加出来的，而是把已有的能力用正确的方式串起来。**
