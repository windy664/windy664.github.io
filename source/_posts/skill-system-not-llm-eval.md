WindyAgent 跑在 Minecraft 服务端上，能查在线、能广播、能跑命令。但服主总会问：「我装了个新插件，能不能让 AI 调它的接口干点啥？」——这就触到了一个所有「会动手的 Agent」迟早要面对的问题：**怎么给它加新能力，又不每次重新编译。**

最性感的答案是「让 LLM 现场写代码执行」。我想了想，决定不这么干。这篇讲为什么，以及我最后选的方案。

<!-- more -->

## 为什么不让 LLM 现场写代码

「让模型按需生成一段脚本、丢进 GroovyShell 跑」听起来很酷，但它和我做这个项目的底层取向是冲突的：

- **不可审**：每次执行的代码都是新的、临时的，你没法提前看一眼它要干什么。在一个能踢人、能改经济数据的生产环境里，这是不可接受的。
- **不确定**：同一个需求，这次生成的代码和下次可能不一样。出了事难复现、难追责。
- **烧 token 还慢**：每次都要现编。

我一直信奉一条线：**能确定化的就别请模型**（之前写过[一篇](/2026/06/13/llm-vs-deterministic/)专门讲这个）。让 LLM 写代码，是把最不该交给它的「确定性执行」也塞了过去。

## 换个思路：服主写好，Agent 按名调用

真正要的不是「LLM 会写代码」，而是「Agent 能调用一段**已经审过的、确定的**能力」。所以方案翻过来：

> 服主自己写好脚本，放进 `skills/` 目录；Agent 只能**按名字调用**，不能创造。审一次、冻结，等于给 Agent 加了一条确定性能力。

服主往目录里放文件这个动作本身，就是信任边界——能往服务器文件系统放东西的人，本来就有完整权限。于是技能不必再过命令护栏，但每次调用照样记审计。

免重编译、可审、确定——三个目标一起拿到。

## 对齐 Anthropic 的 Agent Skills：技能有三种形态

做到一半我去翻了 Anthropic 的 Agent Skills 规范，发现我一开始想窄了——我默认「技能 = 一段可执行脚本」，漏了一整类。对齐之后，技能分三态：

**① 纯文字（流程技能）**：一个 `SKILL.md`，正文是一套**操作步骤**，没有脚本。被调用时，工具直接把正文塞回上下文，Agent 读懂后**用现有工具**去办——它不写代码，只是「照着 SOP 做事」。比如「处理退款」：

```markdown
---
name: handle_refund
description: 处理玩家退款/扣费申诉的标准流程；玩家要求退款时使用
---
1. 用 get_balance 查余额，确认账目
2. 用 knowledge_search 查「退款政策」，判断是否符合
3. 符合 → 用 run_command 执行 eco take，并私聊告知
4. 不符合 → 礼貌说明，不扣减
⚠️ 单笔 > 1000 必须人工审批
```

这正是「渐进式披露」：平时只占一行描述，真用到了才把完整流程加载进来。

**② 脚本 + 文字**：文件夹里 `SKILL.md` 说清「何时用、怎么用」（进工具描述给 LLM 选择），配一个 `script.groovy` 真正干活——做现有工具做不到的事，比如调某个插件的 API 发礼包。

**③ 纯脚本**：扁平一个 `.groovy`，头部用注释声明元数据，兼容最早的写法。

关键在于：**①根本不需要代码**。很多「新能力」其实只是「把已有工具按某套规矩组合起来」，写成一段流程文字就够了，比写脚本安全得多。我之前差点把所有技能都做成「可执行」，是个认知盲区。

## 参数也要先校验，别进 GroovyShell 才报错

脚本技能可以声明参数（`player: string`、`coins: int`）。Agent 填参可能填错或漏填。我在**跳进脚本引擎之前**加了一道预校验——缺参、类型不符直接拦下，回报给 LLM 让它补参，根本不进 GroovyShell：

```kotlin
fun run(def: SkillDef, args: Map<String, Any?>): String {
    def.validate(args)?.let { throw IllegalArgumentException("参数校验失败：$it") }
    return doRun(def, def.script!!, args)   // 过了校验才执行
}
```

这步收口在引擎入口，本地调用和跨服调用两条路都自动覆盖，不用各写一遍。

## 一个让我反复掂量的边界

脚本在主线程执行（Bukkit API 非线程安全），我配了超时看门狗。但看门狗只能**解除 Agent 这边的等待**，没法强杀一个正在主线程上空转的脚本——强行中断主服务器线程会把服务器状态搞坏。

所以真正的兜底从来不是「引擎硬隔离」，而是那条信任边界：**技能由服主编写、已经审过。** 工程上能加的护栏（超时、审计、参数校验）都加了，但最后一道防线是人，不是代码。想清楚这点，整个设计就踏实了。
